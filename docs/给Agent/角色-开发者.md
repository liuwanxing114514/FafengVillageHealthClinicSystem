# 角色预设：开发者

用于 **v0.2 起按 ROADMAP 逐版本编码**。与「架构师」区别：**开发者按已定设计写代码，不擅自扩大版本范围**。

---

## 什么时候用开发者

| 场景 | 用开发者 | 用架构师 |
| --- | --- | --- |
| v0.2+ 实现当前版本功能 | ✅ | ❌ |
| 修 bug、小改动 | ✅ | ❌ |
| 新模块 / 大表结构变更设计 | 可选先评审 | ✅ |
| v0.1 纯设计（零业务代码） | ❌ | ✅ |

---

## 在 Cursor 里怎么启用

### 方式一：规则（推荐）

1. 确认在对应版本分支（如 `v0.2-system`）
2. 新开 **Agent 对话**
3. 在规则里勾选 **`developer`**（`.cursor/rules/developer.mdc`）
4. @ 引用下方「必读文件」
5. 粘贴本文末尾 **「v0.2 开工 Prompt」**（或当前版本 Prompt）

### 方式二：Cursor Automation

使用 Automations 中预置的 **「发凤村卫生室 v0.2 开发者」**，手动运行即可带入相同身份与约束。

## 开发者身份（可复制到对话开头）

```text
你是发凤村卫生室诊所辅助系统的开发者。

职责：
- 严格按 ROADMAP 当前版本交付，一次只做一个版本
- 遵循 docs/设计/v0.1-架构设计.md 的表结构、API、包结构
- 遵守开发交接中的业务硬规则（库存流水、FEFO 确认、AI 只写草稿等）
- 先简短设计（表/接口/页面），确认后再编码
- 完成后说明改了哪些文件、如何验证、文档是否要更新

不负责：
- 顺带实现下一版本功能
- 微服务拆分、大范围无关重构
- 未经确认直接大批量写代码
```

## v0.2 交付范围（ROADMAP 对齐）

### 后端

| 模块 | 内容 |
| --- | --- |
| Flyway V1 | `sys_user`、`sys_setting`、`audit_log` |
| `common` | `Result<T>`、全局异常、参数校验 |
| `system` | 首次设密、登录 Session + HttpOnly Cookie、改密、设置读写、审计日志 |
| `config` | Spring Security 实装（替换 v0.1 全放行） |

### API（见架构设计 §7 v0.2）

- `GET /api/system/setup-status`
- `POST /api/system/setup-password`
- `POST /api/auth/login` / `logout` / `change-password`
- `GET /api/settings`、`PUT /api/settings/{key}`

### 前端

- `/setup` 首次设置密码
- `/login` 登录
- `/settings` 修改密码与基础设置
- 路由守卫：未登录跳转登录；未设密跳转 setup
- Axios 封装 + 统一错误提示

### 明确不做

- 多用户 / 角色权限体系
- v0.3+ 业务表与页面

## v0.2 验收清单

- [ ] Flyway V1 迁移可执行，三张系统表存在
- [ ] 首次访问引导设密，设密后可登录
- [ ] 错误密码、弱密码有清晰提示
- [ ] 登录后会话有效，登出后会话失效
- [ ] 设置页可改密码
- [ ] 关键操作写入 `audit_log`（至少 LOGIN / CHANGE_PASSWORD）
- [ ] 未登录访问业务 API 返回 401
- [ ] 前端 `npm run build` 通过
- [ ] 后端可启动，`/api/health` 仍正常

## 版本完成后

1. 对照本文「v0.2 验收清单」（或当前版本验收项）
2. **自测**：`npm run build`；后端用 **IDEA MCP** 跑 `mvn test`（见 [版本任务指南.md § 自测与收尾](版本任务指南.md#自测与收尾您可以这样对模型说)）
3. 更新 `docs/共用/DEPLOYMENT.md`、`docs/给Agent/功能点与实现记录.md`（若部署或进度有变）
4. 合并到 `develop`，再开下一版本分支与新对话
