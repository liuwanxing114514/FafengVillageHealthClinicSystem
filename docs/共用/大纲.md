# 诊所 AI 辅助系统 - 项目开发大纲

## 一、项目目标

为家庭诊所开发一套本地化 AI 辅助系统，帮助医生减轻药品库存管理、病历记录和处方整理的日常负担。

系统以“核心业务本地运行、数据本地持久化、低维护成本、可长期迭代”为核心原则。第一版同时完成库存管理与门诊病历/处方基础能力，AI 能力做成可拔插模块，默认可接入 DeepSeek API，后续可替换为本地模型。

项目同时带有练手和作品展示性质，需要在架构上体现模块拆分、AI Provider 抽象、Agent 工具调用、人工确认闭环等能力，方便后续作为个人项目写入简历。

开发与运维细节需要完整记录，便于后续维护、迁移和升级。

## 二、硬件与部署环境

- **主力服务器**：Windows 10/11 主力机（AMD R5 3600，16GB DDR4，500GB SSD + 机械盘）。
- **运行环境**：Windows + WSL2 + Docker Compose。
- **服务管理**：优先使用 Docker Compose 管理后端、数据库、前端和后续可选 AI 服务。
- **开机自启**：通过 Windows 任务计划或 NSSM 启动 Docker Compose，而不是单独把 Java 后端注册成 Windows 服务。
- **备用硬件**：群晖 DS920+ 可作为未来迁移或备份设备。

说明：项目采用容器化部署后，后端、数据库和 AI 服务应尽量由 Docker Compose 统一编排，减少“部分服务跑 Windows、部分服务跑 WSL2”的维护复杂度。

## 三、技术栈

### 3.1 后端

- Java 21 LTS
- Spring Boot 3.x
- Spring Web MVC
- Spring Security（简单 PIN / 密码访问保护）
- Spring Validation
- MyBatis-Plus
- Flyway 数据库迁移
- Spring AI 或自定义 AI Client 抽象（用于 DeepSeek API / 本地模型适配）

后端使用 Spring Boot 默认内嵌 Tomcat，并启用 Java 21 虚拟线程：

```yaml
spring:
  threads:
    virtual:
      enabled: true
```

说明：不再同时使用 Undertow 和 Tomcat。第一版优先使用 Spring Boot 默认 Tomcat，配置简单、文档完善，后续维护成本更低。

### 3.2 数据库

- PostgreSQL 16
- pgvector 扩展（远期 RAG / 相似病例检索使用）
- Flyway 管理数据库结构变更

说明：项目统一使用 PostgreSQL，不再使用 `clinic.db` 或 `.db` 表述。业务数据、库存流水、病历、处方、系统配置都存入 PostgreSQL。向量检索能力通过 pgvector 预留。

### 3.3 前端

- Vue 3
- TypeScript
- Vite
- Element Plus
- Pinia
- Vue Router

### 3.4 本地 AI 能力

- **OCR**：PaddleOCR，用于发票、送货单、清单等文字识别。
- **语音转写**：Whisper.cpp，用于病历、处方和通用文本输入。
- **LLM Provider**：默认接入 DeepSeek API，用于结构化整理、病历候选、自然语言查询和 Agent 推理。
- **本地 LLM 备选**：远期可选 Ollama / llama.cpp / 本地 Qwen 模型，用于替换 DeepSeek API。

说明：条码识别不使用 OCR。扫码枪按键盘输入处理；手机可以作为移动扫码枪使用，通过手机浏览器调用摄像头识别药盒条码，再把条码提交给后端。手机扫码使用前端摄像头 + 条码识别库，例如 ZXing。

### 3.5 AI 模块可拔插设计

AI 能力不直接写死在业务模块中，而是通过接口隔离：

- `AiProvider`：统一封装聊天、结构化抽取、摘要、向量化等能力。
- `DeepSeekAiProvider`：默认实现，调用 DeepSeek API。
- `LocalAiProvider`：预留实现，后续接 Ollama / llama.cpp / 本地模型。
- `NoopAiProvider`：降级实现，AI 不可用时系统仍可运行基础业务。

AI 配置集中在 `.env` 或 `application.yml`：

```yaml
clinic:
  ai:
    enabled: true
    provider: deepseek
    api-key: ${DEEPSEEK_API_KEY}
    base-url: https://api.deepseek.com
```

涉及患者姓名、电话等敏感信息时，默认先脱敏再发送给外部 API。AI 返回内容必须进入待确认状态，由医生确认后才能写入正式库存、病历或处方。

### 3.6 备份与运维

- Docker Compose 编排服务。
- PowerShell 备份脚本 `backup.ps1`。
- Windows 任务计划每日定时执行备份。
- 数据库通过 `pg_dump` 导出，不再按 SQLite `.db` 文件备份。

## 四、模块拆分

项目按业务边界拆分模块，便于开发、测试和后续简历展示。

### 4.0 架构拆分原则

第一版不采用微服务架构，采用“模块化单体 + 可插拔外部能力”的形式。

原因：

- 诊所部署环境简单，单机或局域网使用为主，微服务会显著增加部署和维护成本。
- 库存、病历、处方之间存在较多关联，第一版放在同一个后端应用中更容易保证事务一致性。
- 项目需要可维护、可交付、可展示，而不是为了形式拆服务。
- 模块化单体仍然可以清晰体现工程能力：包结构清楚、接口隔离、模块间依赖受控、测试边界明确。

推荐拆分方式：

- **一个后端应用**：包含基础模块、库存模块、病历模块、AI 辅助模块、Agent 模块。
- **一个前端应用**：按页面和业务域拆分目录。
- **一个数据库**：PostgreSQL，按业务域拆表，可使用 schema 或表名前缀区分。
- **可选外部服务**：PostgreSQL、PaddleOCR、Whisper、本地 LLM 可通过 Docker Compose 独立容器运行。
- **外部 API**：DeepSeek API 作为 AI Provider 接入，不作为本地微服务。

后续如果项目变大，可以优先把 AI/OCR/语音这类计算型能力拆成独立服务；库存、病历、处方核心业务暂时不拆微服务。

### 4.1 基础模块

- 访问保护：简单 PIN / 密码。
- 系统配置：端口、数据库、AI Provider、备份路径等。
- 审计日志：记录关键操作。
- 密码管理：首次使用时由用户设置固定密码，后台管理员可修改。

### 4.2 药品库存模块

- 药品资料。
- 入库、出库、盘点修正。
- 库存流水。
- 库存不足与临期预警。
- 条码维护、扫码枪输入和手机扫码输入。
- 支持进货单价。
- 支持拆零单位，例如盒、瓶、颗、支、片等。
- 支持整数单位换算关系，例如 1 盒 = 多少片、1 瓶 = 100 颗，便于拆零出库和库存统计。

### 4.3 门诊病历模块

- 患者管理。
- 病历录入。
- 历史病历查询。
- 处方记录。
- 处方打印。
- 患者、处方、药品之间建立关联，处方药品可生成待出库清单。

### 4.4 AI 辅助模块

- 发票 / 送货单结构化整理。
- 手机拍摄进货清单后生成待审核入库表格。
- 病历语音转写结果整理。
- 常用文本候选。
- 诊疗文本摘要。
- AI 输出待确认草稿。

### 4.5 Agent 模块

- 自然语言查询。
- 工具调用编排。
- 待确认动作生成。
- Agent 执行日志。

Agent 只能生成建议和待确认动作，不直接执行库存扣减、病历保存、处方打印等关键操作。

## 五、阶段规划

> **版本号以 `ROADMAP.md` 为准。** 下表为大纲章节与路线图版本的对应关系。

| 大纲阶段 | 路线图版本 | 说明 |
| --- | --- | --- |
| 第一阶段 MVP | v0.1–v1.0 | 基础业务，不依赖 AI |
| 第二阶段 AI 辅助 | v1.1–v1.3 | 语音、快捷语、DeepSeek 整理 |
| OCR 入库 | v1.4 | 拍照识别打印版单据 |
| 业务扩展 | v1.5–v1.6 | 批量出库、处方模板与就诊收费 |
| 第四阶段 Agent | v2.0–v2.1 | 自然语言查询、流程自动化 |
| 第三阶段 RAG | v2.2–v2.3 | 脱敏向量化、相似病例检索 |
| 第七阶段 AI 运维 | v2.4、v2.6 | Embedding 管理、导出与病历 PDF（v2.5 本地 LLM 已取消） |
| 可部署基线 | v3.0 | AI 全功能整合验收与部署 |

### 第一阶段：库存 + 病历基础版（MVP，v0.1–v1.0）

第一阶段目标是让诊所先真正用起来：药品能建档、能入库、能出库、能查库存、能预警；患者能建档、病历能录入、处方能打印；系统有简单访问保护、备份恢复和基础文档。

第一版开发优先级：门诊功能和药品管理优先（v0.3–v0.5），其次库存流水与预警（v0.6），再补扫码与 Excel（v0.7–v0.8），最后 AI 接口预留与部署（v0.9–v1.0）。

### 5.1 药品基础资料

- 新增、编辑、停用药品。
- 字段包含：药品名称、通用名、剂型、规格、单位、生产厂家、条码、进货单价、**建议零售价**、默认库存下限、备注。
- 支持按名称、拼音首字母、条码搜索。
- 支持同一药品多规格管理。
- 没有现成条码表时，第一版支持手动维护条码。
- 系统提供标准药品 Excel 导入模板，用户按固定表头填写后导入。
- 支持拆零单位，例如盒、瓶、颗、支、片等。库存数量和处方数量都需要明确单位。
- 支持维护整数单位换算关系，例如 1 盒 = 12 片、1 盒 = 10 支、1 瓶 = 100 颗。
- 液体药品暂不按半瓶或毫升拆零，`100ml` 等容量信息只记录在规格中，库存仍按瓶等实际单位管理。
- 默认库存下限为 5 盒（**按基本单位存储**，界面可显示约等于几盒），可按药品单独调整。

### 5.2 入库

#### 5.2.1 手动入库（第一版必做）

在电脑或手机浏览器中打开入库表单，填写药品、规格、数量、单位、批号、有效期、进货单价、供应商等信息，保存后生成入库流水并增加库存。

#### 5.2.2 扫码入库（v0.7）

支持扫码枪和手机扫码两种方式。

- **扫码枪**：扫描药盒条码后，系统像接收键盘输入一样获得条码，自动匹配药品。
- **手机扫码**：手机浏览器打开系统页面，调用摄像头扫描药盒条码，识别后提交给系统。

系统匹配药品后，弹出数量和批号/有效期输入框，确认后入库。

#### 5.2.3 手机拍进货清单 / 发票 / 送货单入库（v1.4）

使用手机拍摄打印版进货清单、发票或送货单，或从电脑上传图片。PaddleOCR 提取文字后，由 DeepSeek API 或本地模型整理为候选入库表格。用户逐条核对、修改、删除或补充后，点击批准入库，系统才生成正式入库流水并增加库存。

AI 识别结果只能进入“待确认草稿”，不得自动直接写入正式库存。

#### 5.2.4 标准 Excel 模板导入（v0.8）

系统提供固定格式的 Excel 模板，用户按模板表头填写药品或初始库存数据后导入。系统导入前先展示预览和错误提示，用户确认后才正式写入。

建议模板字段：

- 药品名称
- 通用名
- 剂型
- 规格
- 基本单位
- 包装单位
- 换算关系
- 生产厂家
- 条码
- 进货单价
- 库存下限
- 批号
- 有效期
- 初始库存数量
- 备注

### 5.3 出库

#### 5.3.1 手动出库（第一版必做）

在药品列表页搜索药品，点击「出库」，输入数量和单位后扣减库存，并生成出库流水。出库时需要关联患者和处方。系统按 FEFO 推荐出库批次，**用户确认批次后**才扣减（可改选其他批次）。

#### 5.3.2 扫码出库（v0.7）

支持扫码枪和手机扫码两种方式。

- **扫码枪**：适合在电脑旁快速出库。
- **手机扫码**：适合药品在柜台外、库房里或不方便使用扫码枪时移动操作。

系统识别条码并匹配药品后，输入数量并确认扣减库存。

#### 5.3.3 批量出库（v1.5）

支持一次选择多种药品并批量扣减库存，可用于盘点损耗、集中领用、整批处理等场景。

### 5.4 库存流水

- 每次入库、出库、盘点修正都记录流水。
- 流水字段包含：药品、批号、有效期、单位、变更前数量、变更数量、变更后数量、操作类型、关联患者、关联处方、操作时间、备注。
- 支持按药品、日期、操作类型查询。

### 5.5 库存预警

- 首页显示库存不足药品。
- 首页显示临近有效期药品。
- 每种药品支持单独设置库存下限。
- 默认库存不足阈值为 5 盒，可配置。
- 默认有效期提前 3 个月提醒，可配置。

### 5.6 盘点与修正

- 支持手动盘点修正库存。
- 盘点修正必须填写原因。
- 盘点修正同样进入库存流水，方便追溯。

### 5.7 患者管理

- 记录患者姓名、性别、**身份证号**、**出生日期**（可选）、年龄、住址、联系方式、备注。
- **年龄**：有身份证号则从证号解析出生日期并推算；有出生日期则据此推算；都没有则手填。
- 支持按姓名、电话、身份证号搜索。
- 支持查看患者历史就诊记录。

### 5.8 病历录入

- 记录主诉、现病史、**既往史**、体温、血压、血氧、呼末二氧化碳、心率、脉象、过敏史、诊断、处理意见、处方等字段。
- **v1.6 起**：每条就诊记录增加 **应收**、**实收**；默认应收 = 关联处方 Σ(建议零售价×数量)，可手改；参考进货成本仅界面展示。
- **v1.6 起**：左侧菜单「病历」；全量列表、直接新建、弹窗快速建患者（不丢已填病历）。
- 支持保存、编辑、查询。
- 支持按患者、日期、诊断搜索。

### 5.9 处方打印

- 第一版生成通用 A4 处方模板（`default-a4`，默认）。
- 处方第一行显示“发凤村卫生室处方签”。
- 处方需要包含患者姓名、性别、年龄、住址、电话。
- 处方药品明细需要包含剂型、药品名称、规格、数量、用法。
- 支持浏览器打印。
- **v1.6**：按真实预印处方纸叠加打印（`preprinted-fafeng`）；可配置字段位置与字号；保留 A4 通用模板为默认；预印模式费别/卡号/处方金额/调剂留空，科别固定「全科」，门诊病历号只印**数字**（无英文标签）。

### 5.10 访问保护

- 第一版增加简单 PIN / 密码。
- 默认单用户模式。
- 首次使用时由用户设置固定密码。
- 后台管理员可修改密码。
- 关键操作需要二次确认，例如删除、批量出库、恢复备份。

### 第二阶段：AI 辅助增强（v1.1–v1.3）

#### 5.11 语音输入（v1.1）

点击字段旁的语音按钮后开始录音，Whisper.cpp 转写为文字，填入当前字段。医生人工核对后保存。

语音转写结果不得绕过人工确认直接写入正式病历。

#### 5.12 快捷语候选（v1.2）

- 按字段维护常用文本候选，例如主诉、诊断、处方分别管理。
- 根据医生历史输入频率生成候选。
- 提供快捷语管理页面，支持新增、编辑、删除。
- 定期清理长期未使用的低频候选。

#### 5.13 DeepSeek 结构化整理（v1.3）

- 将语音转写文本整理成主诉、现病史、诊断、处方等字段草稿。
- 将手机拍摄的进货清单、发票 / 送货单 OCR 文本整理成入库草稿表格。
- 将自然语言问题转成系统查询意图。
- 支持提示词模板管理，便于调试和展示。

### 第三阶段：RAG 辅助知识库（v2.2–v2.3）

#### 5.14 脱敏历史病历入库（v2.2）

去除姓名、电话等隐私信息后，将历史病历文本向量化并存入 PostgreSQL + pgvector。

#### 5.15 相似病例检索（v2.3）

在病历录入页面，根据当前症状检索 Top-3 相似历史病例，仅供医生参考，不替代诊断。

#### 5.19 RAG 运维与 Embedding 管理（v2.4）

系统设置页展示向量化同步状态，支持一键全量/增量同步，便于诊所启用 RAG 后维护向量库。

#### 5.20 导出与病历 PDF（v2.6）

库存流水按筛选导出 Excel；病历页支持保存 PDF（处方页 v1.6 已有同类能力）。

#### 5.21 本地 LLM 接入（v2.5）

实装 `LocalAiProvider`，通过 Ollama / LM Studio 等 OpenAI 兼容接口在断外网环境下使用 Chat 与 Embedding。

### 第四阶段：AI Agent 应用落地（v2.0–v2.1）

#### 5.16 自然语言查询（v2.0）

通过文本或语音查询库存、临期药品、患者历史病历等。Agent 理解意图后调用系统内部工具，例如 `searchMedicine`、`queryInventory`、`searchPatientVisit`。

#### 5.17 简单流程自动化（v2.1）

支持“开处方 -> 自动生成待扣库存清单 -> 医生确认 -> 扣减库存 -> 打印处方”等流程。

库存扣减必须有明确确认步骤，避免 AI 自动误操作。

#### 5.18 作品展示重点

- 展示模块化架构：业务模块、AI 模块、Agent 模块解耦。
- 展示 AI Provider 可拔插：DeepSeek API 与本地模型可切换。
- 展示工具调用：Agent 通过受控工具访问库存、患者和病历 API。

### 7.1 推荐目录结构

```text

  frontend\

  docs\

    给人看\

  exports\

  logs\
  backup\

D:\ai-models\
  paddleocr\
  whisper\
  llm\
```

### 7.2 Docker Compose 服务

第一版建议包含：

- `postgres`：PostgreSQL 数据库。
- `backend`：Java Spring Boot 后端。
- `frontend`：前端静态服务，或由后端托管 `dist`。

后续可增加：

- `ocr-service`：PaddleOCR 服务。
- `whisper-service`：Whisper.cpp 服务。
- `llm-service`：本地大模型服务。

DeepSeek API 不需要单独服务，通过后端调用外部 API。

### 7.3 数据备份

备份脚本 `backup.ps1` 每日凌晨 3 点执行。

备份内容：

- PostgreSQL `pg_dump` 导出的数据库备份。
- 上传文件目录 `uploads`。
- 配置文件 `.env` 和必要的部署文档。

备份文件存放于：

```text
D:\clinic-data\backup\
```

备份保留最近 7 天版本。

建议备份压缩包设置密码，尤其是包含病历数据时。

### 7.4 数据恢复

恢复步骤：

1. 停止 Docker Compose 服务。
2. 解压指定日期的备份包。
3. 恢复 PostgreSQL dump。
4. 覆盖 `uploads` 目录。
5. 启动 Docker Compose 服务。
6. 打开系统检查库存、病历、上传文件是否正常。

### 7.5 版本升级与迁移

| 场景 | 操作步骤 |
| --- | --- |
| 常规功能更新 | 备份当前版本 -> 拉取或复制新版本 -> 重新构建镜像 -> 重启服务 |
| 数据库结构变更 | 先备份 -> 启动新版本 -> Flyway 自动执行迁移 |
| 迁移到新机器 | 复制 `clinic`、`clinic-data`、`ai-models` -> 新机器安装 Docker / WSL2 -> 执行 `docker compose up -d` |
| 切换到 Linux / NAS | 调整 `docker-compose.yml` 挂载路径，保持数据目录结构不变 |

## 九、项目约束

- 库存、病历、处方等核心业务数据必须本地持久化。
- 第一版基础业务不得依赖 AI，避免 AI 服务不可用时系统无法使用。
- AI 模块可接入 DeepSeek API，也必须预留本地模型替换能力。
- 调用外部 AI API 前，应尽量脱敏患者姓名、电话等隐私信息。
- AI 识别、整理、推荐结果必须经过人工确认。
- 关键业务逻辑需要测试覆盖，尤其是库存增减、库存流水、有效期预警、数据库迁移。
- 所有配置项集中在 `.env` 或 `application.yml` 中。
- 数据库结构变更必须通过 Flyway 迁移脚本管理。
- 后端基于 Java 21 LTS，并启用虚拟线程。
- 项目文档必须足够清楚，方便后续维护和迁移。

## 十、已确认决策

1. 数据库确定使用 PostgreSQL。
2. 第一版增加简单 PIN / 密码访问保护。
3. 第一版使用通用 A4 处方模板，第一行显示“发凤村卫生室处方签”。
4. 当前没有药品条码历史表，第一版支持手动维护条码。
5. 第一版同时做库存管理和门诊病历/处方基础功能。
6. AI 模块使用可拔插设计，默认接入 DeepSeek API，后续可替换为本地模型。
7. 项目带有练手和作品展示性质，需要保留清晰架构文档和可展示的 Agent 应用闭环。
8. 第一版不做微服务，采用模块化单体；AI/OCR/语音能力保留后续独立服务化空间。
9. 药品需要记录进货单价。
10. 出库需要绑定患者和处方，患者表、处方表、药品表需要有关联。
11. 支持药品拆零和多单位，例如颗、支、片、盒、瓶等。
12. 病历需要增加体温、血压、血氧、呼末二氧化碳、心率、脉象等字段。
13. 临期药品提前 3 个月提醒。
14. 平时主要使用台式机操作。
15. 第一版最着急优先做门诊和药品管理。
16. 库存不足默认阈值一般为 5 盒，并且需要可配置。
17. 拆零单位之间需要支持换算关系。
18. 简单密码由用户首次设置，保持固定使用，后台管理员可以修改。
19. 需要支持通过手机拍摄进货清单生成待审核入库表格，人工批准后入库。
20. 暂时没有现成药品 Excel 表需要导入，但后续可能会有。
21. 单位换算只需要整数换算，例如 1 瓶 = 100 颗；液体药品不按半瓶或毫升拆零，容量只写入规格。
22. 手机拍进货清单只需要识别打印版清单，不需要识别手写清单。
23. Excel 导入采用系统提供的固定格式模板，用户按模板表头填写内容。
24. T1 脱敏规则（外部 API 出站前）：姓名留姓名用 `*`；手机留前3后4；住址留到村/组/路/街、屏蔽门牌；身份证留前6后4；OCR 单据上联系人/电话/门牌/身份证同样脱敏；药品与诊疗正文可发。
25. T2 出库批号：系统 FEFO 推荐默认批次，用户手动确认后可改选，不自动扣批。
26. T4 年龄：有身份证号或出生日期则自动推算；都没有则手填。
27. T5 第一版患者资料需要身份证号；出生日期可选（可从身份证号解析）。
28. T6 库存下限按基本单位存储，界面显示约等于几盒。
29. 药品资料第一版不追加额外字段。
30. 病历增加「既往史」字段；其余病历字段第一版够用。
31. PostgreSQL 扩展 `vector`（pgvector）、`pg_trgm` 在 v0.1 首条 Flyway 迁移启用；镜像用 `pgvector/pgvector:pg16`。
